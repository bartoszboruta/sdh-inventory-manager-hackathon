# Setup Guide (T3 + Bun + SQLite + tRPC)

## Goal
Bootstrap SDH Inventory with the agreed V1 stack and conventions.

## Prerequisites
- Bun installed (`bun --version`)
- Git installed
- Node-compatible environment for Next.js tooling

## 1) Bootstrap project
From parent folder, run:

```bash
bun create-t3-app@latest --CI --nextAuth --tailwind --drizzle --dbProvider sqlite
```

Expected outcomes:
- Next.js + TypeScript project scaffolded
- NextAuth wired by template
- Drizzle + SQLite baseline config present
- Tailwind configured
- tRPC initialized using T3 conventions

## 2) Install and initialize shadcn/ui
Inside the generated app directory:

```bash
bunx --bun shadcn@latest init
```

Recommended init choices:
- TypeScript: yes
- Style: New York (or default, but keep consistent team-wide)
- Base color: Slate
- CSS variables: yes
- Components path: `src/components`
- Tailwind config/global css paths: use detected defaults

## 3) Add initial UI components
```bash
bunx --bun shadcn@latest add button input form card table dialog badge sheet
```

These are the baseline components for dashboard, CRUD forms, and status badges.

## 4) Confirm folder conventions
Keep this structure:
- `src/app/*` -> frontend routes/pages/layouts
- `src/features/*` -> product-domain frontend modules
- `src/components/ui/*` -> shadcn primitives
- `src/server/api/routers/*` -> backend tRPC routers
- `src/server/db/schema/*` -> Drizzle schema per domain

## 5) Configure SQLite location
Use env variable in `.env` (or `.env.local`) and wire it to Drizzle config.

Example (adjust naming to generated template):
```env
DATABASE_URL="file:./db.sqlite"
```

Notes:
- Keep SQLite file inside project for local dev.
- Do not commit production secrets; for local SQLite only, URL can be committed in example env.

## 6) Generate first migration and apply
Use the script names generated by T3; if custom scripts are added, standardize to:

```bash
bun run db:generate
bun run db:migrate
```

If template uses `drizzle-kit push` workflow, document and keep one approach only.

## 7) Add seed stub (minimum)
Create a seed entrypoint that can insert:
- one company
- one office manager
- one employee
- one office/floor/room
- 2-3 sample assets

Script target convention:
```bash
bun run db:seed
```

## 8) Verify local quality gates
Run:

```bash
bun run dev
bun run lint
bun run typecheck
bun run test
```

Acceptance:
- App starts locally without runtime errors
- Lint and typecheck pass
- Test runner executes unit/integration baseline

## 9) First backend contracts to establish
In `src/server/api/trpc.ts`:
- `protectedProcedure`
- `managerProcedure` (role guard)

In shared types/schema:
- `UserRole` enum with `EMPLOYEE` and `OFFICE_MANAGER`
- Tenant-scoped model convention with `companyId`
- List/pagination response generic for consistent tables and dashboard lists

## 10) Consistency checks before feature work
- Only tRPC API surface in use (no ad-hoc REST endpoints)
- All tenant-owned queries scoped by `companyId`
- Frontend code lives in `src/app` + `src/features`; backend code in `src/server`
- shadcn components are reused instead of ad-hoc duplicates
